## Verdict

Use the existing native repo as a salvageable base, not as an unquestioned continuation.

Do not start from scratch today. The native repo is a real Expo / React Native app with useful routes, Android project files, EAS config, Supabase integration, OneSignal packages, and existing feature work. It now passes `npm run typecheck` and `npx expo-doctor`.

But do not blindly continue the old implementation either. Phase 0 found several P0/P1 issues that must be normalized before broad feature work:

- the native repo has a local stabilization baseline commit, but it is not pushed to origin yet
- Android release signing source of truth must be decided because generated `android/` is ignored by git
- local domain logic duplicates web/shared-package truth
- notification permission is initialized too early
- direct Supabase screen queries need RLS/API contract audit
- storage/audio/navigation patterns need modernization before scale

## Findings

### P0

#### Android release signing source of truth is unresolved

Evidence:

- Before the local stabilization fix, `/Users/Business(C)/shoonaya-mobile/android/app/build.gradle` defined only debug signing and assigned `signingConfig signingConfigs.debug` to release.
- `android/` is ignored by the native repo and is not part of the committed source baseline.
- A local generated-folder fix now defines a release signing config only when explicit `SHOONAYA_UPLOAD_*` Gradle properties are present, but this does not become durable unless Android is tracked or regenerated through config plugins.
- `./gradlew :app:tasks --quiet` parses successfully after the change.

Impact:

- Release credentials still need to be configured before any production or Play internal-track submission.
- If `android/` remains ignored/generated, signing policy must live in EAS credentials and config plugins, not a local ignored Gradle patch.

Required correction:

- Decide whether Android is fully generated by EAS/prebuild or manually maintained.
- Configure EAS/Play signing ownership and keystore custody.
- Verify an EAS Android production build with intended credentials.

#### Native baseline is not pushed

Evidence:

- Native repo has local commit `707f35f chore: stabilize native sdk baseline`.
- Native repo is still ahead of `origin/main`.

Impact:

- Claude, Antigravity, Codex, or a human engineer could audit/build against different source states.
- Remote agents and EAS will not see the baseline unless it is pushed.

Required correction:

- Push or intentionally rebase/squash the local native commit stack before feature migration.
- Record the accepted baseline SHA before assigning Phase 1 prompts.

### P1

#### Notification permission is requested during initialization

Evidence:

- `/Users/Business(C)/shoonaya-mobile/app/_layout.tsx:49-60` calls `initOneSignal()` after fonts load.
- `/Users/Business(C)/shoonaya-mobile/lib/notifications.ts:29-34` initializes OneSignal and calls `OneSignal.Notifications.requestPermission(false)`.

Impact:

- Permission timing can happen before clear user intent.
- This conflicts with the native planning rule that permission prompts must be contextual.

Required correction:

- Initialize OneSignal without prompting.
- Request permission only from onboarding/settings/reminder flows.
- Keep `requestNotificationPermission()` as the explicit prompt path.

#### Domain logic is duplicated locally instead of shared

Evidence:

- `/Users/Business(C)/shoonaya-mobile/lib/panchang.ts:1-55` contains local Panchang astronomy logic.
- `/Users/Business(C)/shoonaya-mobile/lib/pathshala-lessons.ts:1-35` contains local Pathshala lesson pools.
- `/Users/Business(C)/shoonaya-mobile/lib/vrat-data.ts:1-35` contains local Vrat data.
- `/Users/Business(C)/shoonaya-mobile/lib/dharm-veer.ts:1-35` contains local Dharma Veer corpus and daily selection logic.

Impact:

- Native and PWA behavior will drift.
- Content corrections, Dharm Veer rotation, Panchang fixes, and Pathshala source trust will need duplicate updates.

Required correction:

- Replace Panchang and Pathshala local truth with shared packages/API contracts.
- Replace Dharma Veer local seed logic with the DB-backed source of truth already planned.
- Decide whether Vrat/Ekadashi is API-backed or shared package-backed before expanding native Vrat.

#### Direct Supabase screen queries need RLS/API contract audit

Evidence:

- `/Users/Business(C)/shoonaya-mobile/app/mandali.tsx` directly reads/writes `profiles`, `posts`, `post_upvotes`, `post_reports`, and `user_blocks`.
- `/Users/Business(C)/shoonaya-mobile/app/(tabs)/tirtha.tsx` directly writes `tirtha_places`, `tirtha_saves`, and `tirtha_checkins`.
- `/Users/Business(C)/shoonaya-mobile/app/settings.tsx:83-126` reads/writes `profiles`.
- `/Users/Business(C)/shoonaya-mobile/lib/supabase.ts:13-24` uses the public client directly in native.

Impact:

- This can be correct only if RLS is complete and tested for each table.
- Some operations may require API routes if they rely on moderation, service role, aliasing, entitlement, or trusted geodata.

Required correction:

- Create a native data access matrix before touching these features.
- Confirm RLS for every direct native table.
- Move service-role or moderation-sensitive writes behind API routes.

#### Android permissions are broad relative to current feature certainty

Evidence:

- `/Users/Business(C)/shoonaya-mobile/app.json:21-31` declares location, camera, record audio, boot completed, and audio settings permissions.
- `/Users/Business(C)/shoonaya-mobile/android/app/src/main/AndroidManifest.xml:2-12` also contains storage, system alert window, vibration, location, camera, and record audio permissions.

Impact:

- Play data safety and permission review become harder.
- Users may see scary permission surfaces before feature value is clear.

Required correction:

- Prune permissions to actual Phase 1 needs.
- Add permissions only when the corresponding native feature ships.
- Validate Android 13 notification and Android 14 foreground-service implications.

#### Native storage and audio choices need modernization

Evidence:

- `/Users/Business(C)/shoonaya-mobile/lib/supabase.ts:3-24` stores Supabase sessions in AsyncStorage.
- `/Users/Business(C)/shoonaya-mobile/app/settings.tsx:14-137` stores settings/theme in AsyncStorage.
- `/Users/Business(C)/shoonaya-mobile/hooks/useAudioPlayer.ts:1-25` uses `expo-av`.
- Native Expo guidance prefers `expo-audio` over `expo-av`; current `package.json` excludes `expo-av` from doctor validation instead of migrating.

Impact:

- Session storage may be less appropriate than SecureStore-backed storage.
- Audio work may carry deprecated API risk.
- Future offline Sadhana storage needs a deliberate adapter, not scattered storage calls.

Required correction:

- Decide storage policy: SecureStore for sensitive auth/session material, localStorage/MMKV/SQLite for non-sensitive state.
- Plan `expo-av` to `expo-audio` migration.
- Add a storage adapter before integrating `@sangam/sadhana-engine`.

### P2

#### Navigation works but is not yet aligned with native-launch hierarchy

Evidence:

- `/Users/Business(C)/shoonaya-mobile/app/(tabs)/_layout.tsx:1-65` uses JS `Tabs` and tab order Home, Pathshala, Bhakti, Tirtha, Profile.
- `PRODUCT_CONSOLIDATION_PLAN.md` says Kul should be a primary launch destination, while Tirtha/Mandali should remain contextual until usage proves otherwise.

Impact:

- Current tab bar is usable but not product-final.

Suggested correction:

- Keep the current shell during stabilization.
- In Phase 1/IA ADR, decide the final tab set and whether Kul replaces Tirtha in primary navigation.

#### UI implementation has web-era/mobile-v1 patterns

Evidence:

- `/Users/Business(C)/shoonaya-mobile/app/(tabs)/_layout.tsx:1-2` uses `@expo/vector-icons`.
- Many screens are large route files, e.g. Mandali 610 lines, Bhakti 623 lines, Panchang 241 lines, Profile 573 lines.
- Native Expo guidance prefers route files as route entry points with supporting components outside `app/`.

Impact:

- This is maintainable for audit, but not ideal for long-term native quality.

Suggested correction:

- Do not refactor everything at once.
- When a screen enters Phase 1/2 implementation, split reusable components out of route files and keep route files as orchestration.

#### Account deletion exists but needs UX/compliance hardening

Evidence:

- `/Users/Business(C)/shoonaya-mobile/app/settings.tsx:140-165` calls `/api/user/delete` through `apiFetch`.

Impact:

- This is a good compliance start.
- It still needs final copy, confirmation UX, session cleanup behavior, and backend contract verification.

Suggested correction:

- Keep the route.
- Add it to the Store Compliance audit checklist.
